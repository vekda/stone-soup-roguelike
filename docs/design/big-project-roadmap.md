# 《地牢炖汤》大项目扩展方向报告

日期：2026-05-11  
项目路径：`C:\Users\vekda\OneDrive\workspace\stone-soup-roguelike`  
当前定位：浏览器传统回合制 roguelike，石汤/地下城探索气质，素材与规则原创。

## 1. 当前状态判断

《地牢炖汤》已经具备一个“小而完整”的 roguelike 原型：

- 随机房间地牢与楼层推进；
- 回合制移动、撞怪攻击、等待、拾取与使用物品；
- 5 个职业，每个职业有主动技能；
- 召唤物、冷却、临时效果、升级、死亡/胜利状态；
- SVG 像素风资产与单文件 HTML 分享能力；
- 核心逻辑有 Node test 覆盖。

朋友测试反馈“难度适中 / 反馈挺好”，这是非常重要的信号：**当前最值得保留的是高压但公平的节奏**。后续扩展不应先削弱怪物或堆更多回血，而应增加：

1. 更多可理解的战术选择；
2. 更多中长期成长目标；
3. 更多关卡变化与敌人行为；
4. 更好的可读性、反馈和重玩动力。

一句话产品判断：

> 这个项目适合扩展成“轻量但深”的浏览器传统肉鸽，而不是扩展成内容堆砌型 RPG。

## 2. 扩展成大项目的核心原则

### 2.1 保持传统 roguelike 的硬核味

应保留：

- 网格；
- 回合；
- 永久死亡；
- 资源压力；
- 不确定性；
- 职业差异；
- 一局 10-30 分钟可结束的节奏。

不要过早加入：

- 大量剧情文本；
- 复杂任务系统；
- 长篇新手引导；
- 重度养成数值；
- 实时动作操作。

### 2.2 难度适中，不要直接下调敌人

朋友反馈说明当前难度曲线不是主要问题。后续如果要优化体验，优先做这些：

- 增加敌人意图提示，而不是降低攻击；
- 增加地形/道具反制手段，而不是减少怪物；
- 增加风险奖励选择，而不是送资源；
- 改善死亡复盘，让玩家觉得“下次能更好”。

### 2.3 每个扩展都要服务“决策密度”

好 roguelike 的乐趣不是内容数量，而是每回合都有小选择。新增内容应回答：

- 玩家是否需要权衡？
- 是否改变站位、资源或风险？
- 是否和职业/道具/敌人产生组合？
- 是否能被测试覆盖？

## 3. 建议的大项目方向：三条路线

### 路线 A：轻量精品版，适合最快做成可分享作品

目标：把现有原型打磨成一个完整、稳定、朋友愿意反复玩的网页小游戏。

适合周期：1-3 周。  
目标体量：10-15 层，8-10 职业，20-30 道具，20 敌人以内。  
重点：手感、平衡、可读性、分享。

新增内容：

- 更多敌人类型；
- 精英怪/小 boss；
- 祭坛/事件房；
- 2-3 个结局；
- 死亡统计和胜利记录；
- 每日种子挑战；
- 单文件版持续维护。

优点：最容易完成，最适合继续给朋友测。  
风险：内容量有限，但可以靠高完成度取胜。

### 路线 B：系统深度版，适合做成长期开发项目

目标：做成可持续扩展的原创 roguelike 系统。

适合周期：1-3 个月。  
目标体量：多分支地下城、职业天赋、物品词缀、怪物生态、局外图鉴。  
重点：架构、数据驱动、平衡工具。

新增内容：

- 职业天赋树；
- 装备词缀；
- 怪物技能和 AI 意图；
- 地牢主题层；
- 状态系统；
- 存档/战绩/图鉴；
- 平衡模拟脚本。

优点：可扩展空间大，后续每次开发都有明确方向。  
风险：需要重构数据模型和 UI，否则内容一多会混乱。

### 路线 C：叙事主题版，适合形成独特风格

目标：围绕“地牢炖汤 / 食材 / 锅 / 香料 / 发酵”打造有记忆点的原创世界。

适合周期：1-2 个月。  
重点：主题统一、事件、角色、怪物命名、轻文本。

新增内容：

- 食材派系；
- 烹饪祭坛；
- 锅炉房、盐沼、发酵窖、古龙汤锅等主题层；
- NPC 厨师/商人/诅咒食评家；
- 食谱收集；
- 不同“汤底”作为局内流派。

优点：差异化强，不只是“另一个地下城肉鸽”。  
风险：如果文本过多，会稀释当前简洁节奏。

## 4. 推荐路线

推荐采用：**A 起步，B 做骨架，C 做主题包装。**

也就是：

1. 先把现有原型升级为可反复测试的 v0.2；
2. 同时做必要的数据驱动改造；
3. 内容扩展都围绕“地牢炖汤”的原创主题。

不要一上来做庞大世界观或完整天赋树。当前最有价值的是“朋友已经觉得好玩”，应趁这个窗口快速迭代可玩内容。

## 5. 具体可做模块

### 5.1 地牢层级与主题区

当前地图是通用房间 + 走廊。大项目应改成多主题层：

| 层段 | 主题 | 机制 | 代表敌人 |
|---|---|---|---|
| 1-3 | 冷灶入口 | 基础教学，少量陷阱 | 地牢鼠、盐沼哥布林 |
| 4-6 | 盐沼储藏间 | 湿滑地面、腐蚀池 | 腌菜怪、盐壳蟹 |
| 7-9 | 发酵窖 | 毒气、孢子、召唤 | 酵母团、霉菌术士 |
| 10-12 | 炖锅核心 | 火焰、蒸汽、爆炸锅炉 | 火汤元素、锅炉傀儡 |
| 13+ | 古龙汤厅 | boss 前置挑战 | 石汤古龙、龙厨祭司 |

开发建议：

- `src/game/map.js` 增加 `theme`；
- `src/game/data.js` 增加 `DUNGEON_THEMES`；
- 怪物和物品根据主题加权生成；
- 每个主题只先做一个机制，避免失控。

### 5.2 敌人 AI 与敌人技能

当前敌人主要是靠近并攻击。可以增加少量可读行为：

| 敌人类型 | 行为 | 玩家对策 |
|---|---|---|
| 弓手/投叉哥布林 | 直线远程攻击 | 躲墙角、逼近 |
| 胀气酵母 | 死亡爆炸 | 控距离击杀 |
| 盾锅兵 | 正面高防，侧后弱 | 走位绕开 |
| 汤巫师 | 给敌人治疗/加防 | 优先击杀 |
| 火汤元素 | 留下燃烧地格 | 改变路线 |

重点是“意图可读”：

- 敌人下回合要施法时显示标记；
- 鼠标/点击敌人显示说明；
- 日志提示“锅炉傀儡开始蓄热”。

不要把敌人 AI 做得太聪明。roguelike 里敌人应该给压力，但玩家要能学习规律。

### 5.3 物品系统升级

当前物品有治疗、装备、卷轴。可以扩展为：

#### 消耗品

- 辣椒油瓶：范围燃烧；
- 冰镇酸梅汤：冻结周围敌人 1 回合；
- 发酵粉：临时召唤小面团；
- 盐袋：解除毒/腐蚀；
- 浓汤：回复但降低速度/下回合不能动。

#### 装备

- 主手：菜刀、汤勺、擀面杖；
- 防具：锅盖、围裙、铁锅盔；
- 饰品：香料包、旧菜单、龙骨汤匙。

#### 词缀

先做少量词缀：

- 锋利：攻击 +1；
- 滚烫：命中附带 1 火伤；
- 稳固：防御 +1；
- 贪食：击杀回血 1；
- 发酵：技能冷却 -1，但最大 HP -2。

架构建议：

- 不要继续把永久装备直接当 `kind: weapon/armor` 处理到底；
- 引入 `equipment` 槽位；
- 物品效果抽象为 `effects` 或 `modifiers`。

### 5.4 职业扩展

当前 5 职业差异已经成立。可以扩到 8-10 个，但每个都要有独立玩法。

候选职业：

| 职业 | 核心机制 | 技能 |
|---|---|---|
| 炉火僧 | 越低血越强 | 炉心爆气：低血时范围击退 |
| 盐术士 | 控制/减速 | 撒盐结界：区域内敌人减速 |
| 菌菇猎人 | 毒与陷阱 | 孢子陷阱：敌人踩中中毒 |
| 汤勺骑士 | 位移/推拉 | 舀击：把敌人推入墙/陷阱 |
| 食评家 | 鉴定/风险奖励 | 毒舌点评：标记敌人，击杀掉落奖励 |

但建议短期只加 1-2 个。职业越多，平衡成本越高。

### 5.5 成长系统

当前升级是自动加生命和攻击。大项目可以做成“三选一升级”：

每次升级给 3 个选项：

- 最大 HP +4；
- 攻击 +1；
- 防御 +1；
- 技能冷却 -1；
- 背包 +1；
- 某职业专属强化。

职业专属例子：

- 铁锅战士：锅盖猛击命中后击退；
- 香料法师：辣雾爆燃范围 +1；
- 菜刀游侠：背刺击杀返还冷却；
- 汤药医师：溢出治疗转护盾；
- 酵母召唤师：最多 2 个面团。

这样既增加 build 感，又不会破坏一局短节奏。

### 5.6 地形、陷阱和交互物

增加少量地形能显著提高战术性：

| 元素 | 效果 |
|---|---|
| 油渍 | 火焰伤害翻倍/可被点燃 |
| 蒸汽孔 | 每隔几回合喷发 |
| 盐池 | 腐蚀防御，但可伤敌 |
| 菜板机关 | 推敌触发尖刺 |
| 祭坛 | 献祭物品换奖励/诅咒 |
| 商人锅 | 消耗金币买道具 |

这类系统要少而精。每种地形必须能被玩家利用，而不是只惩罚玩家。

### 5.7 Boss 与小 Boss

建议每 3 层一个小 boss，每个主题一个 boss：

- 盐沼蟹王：横向冲锋，撞墙眩晕；
- 发酵母巢：不断生小面团，但本体不动；
- 锅炉傀儡：蓄热 2 回合后范围爆发；
- 石汤古龙：多阶段 boss，吐息改变地形。

Boss 设计原则：

- 不靠纯数值压人；
- 招式有预告；
- 玩家能通过走位、物品、技能反制；
- 每个 boss 教会一种机制。

### 5.8 局外系统

如果要让朋友反复玩，可以加轻量局外功能，但不要变成刷数值：

推荐：

- 死亡记录；
- 胜利记录；
- 已遇到怪物/道具图鉴；
- 每日挑战种子；
- 职业通关徽章；
- 最深层数排行榜，本地即可。

不推荐早期做：

- 永久属性加成；
- 付费式成长；
- 长期材料刷取。

局外系统应该增强回忆和目标，而不是降低游戏难度。

## 6. 技术架构建议

当前代码量还小，但已经接近需要分层的节点。建议下一阶段先做“可扩展但不过度工程化”的重构。

### 6.1 数据驱动

建议把 `src/game/data.js` 拆成：

```text
src/game/data/
├── classes.js
├── enemies.js
├── items.js
├── themes.js
├── skills.js
└── index.js
```

后续内容增加时，不要让单个 `data.js` 变成几千行。

### 6.2 引擎模块化

建议把 `src/game/engine.js` 拆成：

```text
src/game/engine/
├── state.js          # newGame / enterDepth
├── combat.js         # attack / damage / death / xp
├── turns.js          # finishPlayerTurn / enemyTurn / waitTurn
├── skills.js         # useSkill + skill handlers
├── inventory.js      # useItem / equipment
├── ai.js             # enemy decision
└── index.js
```

拆分顺序要谨慎：先补测试，再小步移动函数，不要一口气重写。

### 6.3 状态效果系统

现在 `effects` 已有雏形。建议升级为统一结构：

```js
{
  id: 'burning',
  name: '燃烧',
  turns: 3,
  stacks: 1,
  modifiers: { damageReduction: 0, defenseBonus: 0 },
  tick: 'damage:2'
}
```

但不要立刻做脚本解释器。先用简单表驱动和 switch 即可。

### 6.4 测试与平衡工具

当前测试覆盖很好。大项目要增加：

- 地图连通性测试；
- 所有敌人/物品/职业 sprite 存在测试；
- 技能失败不耗回合测试；
- 状态效果持续时间测试；
- boss 招式测试；
- 1000 局自动模拟，统计死亡楼层、职业胜率。

建议新增脚本：

```text
scripts/simulate-runs.mjs
```

输出：

```text
职业 1000 局平均到达层数
死亡原因 Top 5
每层平均剩余 HP
通关率
```

这比凭感觉调数值可靠得多。

## 7. UI/体验提升建议

当前 UI 足够原型，但大项目需要提升可读性。

优先级最高：

1. 敌人 tooltip：名称、HP、攻击、防御、特殊能力；
2. 攻击/受伤浮字；
3. 技能目标范围高亮；
4. 敌人意图图标；
5. 小地图或楼层信息；
6. 游戏结束复盘。

移动端适配可以后做，除非朋友主要用手机玩。

## 10. 结论

《地牢炖汤》最有潜力的方向不是把它做成大而全的 RPG，而是做成：

> 主题鲜明、节奏紧凑、每回合有决策、朋友愿意反复挑战的原创浏览器 roguelike。

当前难度反馈是正面的，所以扩展时应坚持“增加选择，不降低压力”。

最推荐的下一阶段目标是 v0.2：

- 战绩/死亡复盘；
- 敌人信息和意图；
- 少量新敌人能力；
- 少量新物品；
- 第一个小 boss；
- 每日种子；
- 单文件版继续分享测试。

这条路线最稳：既能快速让朋友玩到新内容，又能为长期大项目打下数据和架构基础。
